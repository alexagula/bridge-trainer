# Экспертный анализ: Следующий этап развития Bridge Trainer

**Дата:** 2026-03-28
**Тема:** Определение вектора улучшений после завершения MVP и аудита безопасности

---

## Как я понял тему

Проект «Тренажёр бриджа» — работающее PWA с 12 модулями, задеплоено на GitHub Pages. Прошли этапы: создание, контент, безопасность, баг-фиксы. Нужно определить следующий вектор улучшений, который даст максимальную пользу для обучения при минимальных усилиях.

---

## Эксперты

**Эксперт 1:** Duolingo Learning Design Team (архетип) — геймификация обучения, retention loops
*Почему:* Bridge Trainer — по сути Duolingo для бриджа. Вопрос удержания и мотивации — ключевой для обучающего приложения.

**Эксперт 2:** Dan Abramov — архитектура frontend-приложений, DX, состояние
*Почему:* 7500+ строк vanilla JS без фреймворка — проект на грани, где архитектурный долг начинает тормозить развитие.

**Эксперт 3:** Jeff Patton — продуктовый discovery, story mapping, MVP-итерации
*Почему:* Проект дошёл до стадии «всё работает, что дальше?» — нужен продуктовый взгляд.

**Главный: Jeff Patton** — главный риск сейчас не код, а направление. Без ясного ответа на «что бесит при тренировке?» можно месяцами улучшать то, что не нужно.

## Анализ от каждого эксперта

### Duolingo Learning Design Team — ключевые принципы

1. **Session framing** — Duolingo начинает сессию с цели («5 минут»), показывает прогресс внутри сессии (3/10), и завершает с reinforcement. Bridge Trainer имеет Daily Mix, но нет ощущения «сессии» — задачи идут бесконечно, нет финиша.

2. **Error-driven learning** — Duolingo показывает ошибку, объясняет, и через 2-3 задачи даёт ТУ ЖЕ тему снова. SM-2 в Bridge Trainer работает между сессиями (завтра повторишь), но внутри сессии ошибочная тема не возвращается.

3. **Progressive disclosure** — новичок не должен видеть 12 модулей сразу. Duolingo открывает уровни постепенно. В Bridge Trainer все модули доступны сразу — перегрузка выбором.

4. **Micro-celebrations** — правильный ответ должен ощущаться. Сейчас feedback текстовый (зелёный/красный блок). Нет анимации, звука, ощущения прогресса.

### Dan Abramov — ключевые принципы

1. **BaseTrainer не используется** — создан, но ни один тренажёр его не наследует. Каждый тренажёр дублирует: stats bar, кнопку «Далее», запись в ProgressTracker. Это 12x дублирование одного паттерна.

2. **innerHTML как шаблонизатор** — весь UI строится через template literals в innerHTML. Работает, но: нет реактивности, XSS-риски, обработчики теряются при перерисовке.

3. **State management отсутствует** — состояние тренажёра хранится в разрозненных `this.*` полях. Нет единого state → render цикла. Добавление фич (undo, history, replay) потребует переписать каждый тренажёр.

4. **Lifecycle не гарантирован** — destroy() не всегда чистит таймеры и обработчики. Утечки памяти при переключении модулей.

### Jeff Patton — ключевые принципы

1. **«Что бесит» > «Что можно добавить»** — roadmap перечисляет 20+ фич, но ни одна не проверена на реальном использовании.

2. **MVP-фичи уже сделаны** — 12 тренажёров, SM-2, Daily Mix, прогресс, PWA. Дальнейшее развитие должно быть pull (потребность), не push (идея).

3. **Единственный пользователь = лучший тестировщик** — автор сам учится бриджу. Каждая тренировочная сессия — user test.

## Решение от Jeff Patton

Проект находится в опасной точке: всё работает, и появляется соблазн добавлять фичи «на будущее». Это классическая ловушка — feature factory без обратной связи.

Главная возможность: автор сам является единственным пользователем и новичком в бридже. Это уникальная позиция — ты одновременно product owner и target user.

**Подход: одна неделя тренировок → записать 5 главных friction points → исправить топ-3.** Не планировать больше одного цикла вперёд.

Риски:
- Уйти в рефакторинг архитектуры (BaseTrainer, state management) — не даст пользы для обучения
- Добавить AI-тренера раньше, чем кончились объяснения из decision tree
- Расширять контент до того, как прорешал существующие задачи

В первую очередь: исправить критические баги из предыдущего аудита (getAllStats не видит tricks/defense, утечки таймеров, null guards). Это блокирует нормальную тренировку.

## План действий

| # | Что сделать | Приоритет | Почему именно это | Ожидаемый эффект |
|---|-------------|-----------|-------------------|------------------|
| 1 | Исправить баги из предыдущего аудита (getAllStats не видит tricks/defense, утечки таймеров, null guards) | Высокий | SM-2 не работает для 2 модулей, зомби-таймеры, крэши при переключении | Все 12 модулей стабильно работают |
| 2 | Добавить session framing в Daily Mix — прогресс-бар (3/10), финальный экран с результатами | Высокий | Daily Mix — основной режим, но нет ощущения завершённости | Мотивация завершить сессию |
| 3 | Intra-session error retry — при ошибке через 2-3 задачи показать аналогичную | Высокий | SM-2 повторяет завтра, но ошибку нужно закрепить сейчас | -30% повторных ошибок |
| 4 | Тренироваться неделю, записать friction points | Высокий | Всё в roadmap — гипотезы. Реальное использование покажет, что нужно | Решения на основе данных |
| 5 | Micro-feedback: CSS-анимация на правильный/неправильный ответ | Средний | Текстовый feedback не даёт эмоционального отклика | Тренировка ощущается живой |
| 6 | Скрыть модули с accuracy 0% из tab bar (progressive disclosure) | Средний | 12 модулей сразу = перегрузка выбором | Фокус, меньше паралича |
| 7 | Консолидировать BaseTrainer — stats, next button, record | Низкий | 12x дублирование. Но рефакторинг ≠ пользовательская ценность | Меньше кода, проще поддержка |
| 8 | Экспорт прогресса в JSON | Низкий | localStorage = одно устройство. Пока один пользователь — не критично | Бэкап данных |
